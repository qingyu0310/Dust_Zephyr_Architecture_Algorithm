# 空载开环 `torque -> omega` 辨识为什么不适合

## 目的

本文说明当前仓库里这套电机本体辨识方法，在**空载、纯转矩开环、长时间平台激励**条件下，为什么不适合拿来拟合全局的 `torque -> omega` 传递函数。

这里的“不适合”不是指 RLS 算法失效，而是指**实验工况本身已经把对象推入明显的非线性区**，导致辨识结果只能拟合现象，不能再直接解释为有物理意义的全局传函参数。

---

## 对应代码

- 测试线程: `project/thread/test/trd_test.cpp`
- 在线辨识器: `algorithm/identify/motor/motorplant.hpp`
- 在线辨识实现: `algorithm/identify/motor/motorplant.cpp`
- 当前测试电机反馈: `modules/motors/dji/dji_c6xx.hpp`
- 达妙 MIT 模式说明: `modules/motors/dm/dm.hpp`

当前 `MotorPlant` 使用的模型是：

```text
Δω = Δt * (p1 * ω + p2 * τ(t-L) + p3 + p4 * sign(ω))
```

其中：

- `p1` 对应极点，`tau = -1 / p1`
- `p2` 对应转矩增益，`K = -p2 / p1`
- `p3` 是常值偏置
- `p4` 是 `sign(ω)` 项，用来吸收方向相关的非线性

---

## 当前测试方法

`project/thread/test/trd_test.cpp` 里的测试流程是：

1. 给定一组固定转矩幅值 `amp`
2. 交替发送 `+amp` / `-amp`
3. 每一段保持 `5000` 个周期，每周期 `1 ms`
4. 在线记录 `snap.omega`
5. 用 `MotorPlant` 做 RLS 更新

也就是说，当前实验不是小扰动测试，而是**长时间纯转矩平台激励**。

---

## 实际观测到的现象

在多个幅值下，日志都出现了同一个特点：`max_omega` 几乎不随 `amp` 变化。

例如，曾经测到如下结果：

```text
amp=0.060000  max_omega=35.659340
amp=0.090000  max_omega=35.549454
amp=0.120000  max_omega=35.527473
amp=0.150000  max_omega=35.549450
```

以及较大幅值时：

```text
amp=0.200000  max_omega=36.098900
amp=0.300000  max_omega=38.802197
amp=0.400000  max_omega=39.197803
amp=0.500000  max_omega=40.494507
```

可以看出：

- 小幅值时，电机几乎都会冲到同一个速度上限附近
- 较大幅值时，最终速度仍主要受上限机制支配
- 幅值主要改变“加速多快”，而不是“最后停在哪”

这说明当前数据的主导现象已经不是线性传函，而是**持续加速直到碰到速度上限/内部限制**。

---

## 为什么这套方法不行

### 1. 纯转矩空载开环下，没有想要的稳态

当负载很小、又采用纯转矩开环时，电机不会像一阶线性对象那样“给一个输入后收敛到某个与输入成比例的稳态速度”。

更常见的行为是：

1. 非零转矩一旦超过摩擦阈值，电机开始持续加速
2. 速度不断升高
3. 最终由反电势、电流限制、驱动内部限速、供电限制等因素夹在某个速度附近

因此，输入转矩在这种工况下主要决定的是**加速度**，不是全局意义上的**稳态速度**。

这时再去拟合全程 `torque -> omega` 的开环传函，物理前提就已经不成立。

### 2. 长平台激励让数据被“高速端限幅区”主导

当前每段平台持续约 `5 s`。这远大于前段加速的主要动态时间。

结果是：

- 前面一小段可能还带一点线性动态信息
- 后面绝大部分样本已经进入“高速端被夹住”的区域

RLS 会优先拟合样本最多的那部分数据，因此最终更容易学到：

- “最后都撞到某个速度上限”

而不是：

- “低速局部的线性动态”

### 3. `p2` 和 `p4` 在当前数据里强相关

模型里同时存在：

- `p2 * τ(t-L)`
- `p4 * sign(ω)`

但在长时间纯转矩平台下，常常会出现：

```text
sign(ω) ≈ sign(τ)
τ(t-L) ≈ amp * sign(ω)
```

这意味着 `p2` 项和 `p4` 项在回归向量里几乎共线。

一旦共线，RLS 很难分别辨清：

- 哪一部分应该算进转矩增益 `p2`
- 哪一部分应该算进方向项 `p4`

结果就是：

- `p1` 常常比较稳定
- `tau = -1 / p1` 看起来还不错
- `p2`、`p4` 会互相“背锅”
- `K = -p2 / p1` 会随幅值漂移，失去明确物理意义

### 4. 当前结果更像“现象拟合”，不是“对象传函”

如果所有幅值最后都被夹在接近同一个速度附近，那么模型最终学到的是：

- 一个“加速到上限”的行为描述

而不是：

- 一个可以推广到不同工作点的全局线性对象

这时候参数虽然能把数据拟合得还行，但它们并不能直接代表真实电机的全局 `K`、全局 DC 增益或全局稳态传函。

---

## 达妙 MIT 模式下这个问题更明显

`modules/motors/dm/dm.hpp` 的注释里已经写得很直白：

- 当 `kp = 0, kd = 0` 时，给定 `t_ff` 可以实现给定扭矩输出
- 但在空转或轻载下，如果 `t_ff` 较大，电机会持续加速，直到加速到最大速度

也就是说，对 DM 的 MIT 纯转矩模式来说，这不是偶发现象，而是**官方文档已经明确说明的行为特征**。

因此：

- 对当前 DJI C6xx 空载电流/转矩开环测试，这个问题存在
- 对 DM MIT 纯 `t_ff` 空载测试，这个问题同样存在，而且更典型

---

## 现在还能相信什么

在这类实验下，通常还能保留一些价值的是：

### 1. `tau`

`p1` 往往比较稳定，因此：

```text
tau = -1 / p1
```

可以作为**低速局部动态**或者**速度环初始调参**的参考量。

但这里的 `tau` 仍应理解为：

- 当前工况下的局部时间常数

而不是：

- 覆盖全速域、全载荷的统一真值

### 2. 前段短时加速信息

在阶跃刚开始的前几十毫秒到一两百毫秒内，数据仍然可能包含有用的局部动态信息。

这一小段更适合回答：

- 给定一个小转矩扰动，速度增长有多快

而不适合回答：

- 这个输入最终对应什么稳态速度

---

## 不应该怎么解释当前结果

在当前实验条件下，不应再把下面这些量当成全局物理参数：

- `K = -p2 / p1`
- “空载开环 torque->omega 传递函数的 DC 增益”
- “一个幅值下拟合出来的全局稳态速度关系”

这些量在当前数据里更接近：

- 对某一段测试现象的折中拟合

而不是：

- 可外推、可复用的真实电机本体参数

---

## 推荐替代方案

### 方案 1: 只做前段短时局部辨识

把问题改成低速局部模型：

- 只取阶跃后的前 `20~100 ms` 或 `50~150 ms`
- 只保留低速区数据，例如 `|omega| < 5` 或 `< 10 rad/s`
- 拟合 `torque -> domega/dt` 或局部 `torque -> omega`

这样得到的是小信号、低速、局部线性模型。

### 方案 2: 先加负载，再做开环辨识

如果希望得到更像 `torque -> omega` 的稳态关系，需要让对象在给定转矩下确实能形成工作点，而不是一路加速到上限。

例如：

- 加实际机械负载
- 让粘滞阻力显著增加
- 在真实工作机构中测试，而不是空载悬空

### 方案 3: 换成速度环附近的小扰动辨识

如果最终目的是调速度环或做仿真，更实用的办法通常是：

- 先闭上内层电流/转矩实现
- 再围绕某个速度工作点注入小扰动
- 辨识局部 `speed loop plant`

这样得到的模型虽然不是“裸电机全局本体”，但更贴近最终控制问题。

---

## 结论

结论可以直接写成一句话：

> 空载、纯转矩开环、长时间平台激励下，电机输入转矩主要决定加速过程，而最终速度主要由内部限速/反电势/供电约束决定。  
> 因此，这种工况不适合做全局开环 `torque -> omega` 传递函数辨识。

进一步说：

- `tau` 可以保留为局部动态参考
- `K` 不应再当成全局 DC 增益解释
- 当前结果说明的是实验工况进入了非线性区，不是 RLS 必然失效
